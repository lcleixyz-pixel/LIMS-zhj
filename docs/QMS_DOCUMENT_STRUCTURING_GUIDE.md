# 体系文件结构化说明

本文档说明如何把外部依据、质量手册、程序文件、作业指导书和记录表格转为 Markdown 结构化内容，并在系统层面进行条款级、要素级、块级追溯和版本输出。

## 1. 设计目标

结构化文件不是简单的附件上传，而是把现有文件拆成可追溯、可维护、可组合的体系内容。

目标包括：

- 原始文件固定位置归档，保留文件名、哈希、来源和状态。
- 文件文本转为 Markdown，按章节和控制要求拆为内容块。
- 每个内容块可映射到外部条款、要素、手册章节、程序文件、记录表格、岗位和运行模块。
- 修改单个内容块时可以生成变更日志和影响范围。
- 可以随时把结构化内容重新组合为完整体系文件包。

## 2. 文件角色

| 角色 | `document_role` / `source_kind` | 说明 |
|------|---------------------------------|------|
| 外部依据 | `external_basis` | 标准、准则、补充要求 |
| 质量手册 | `quality_manual` | 实验室质量手册 |
| 程序文件 | `procedure` | 管理和技术程序文件 |
| 作业指导书 | `work_instruction` | 操作规程、作业指导 |
| 记录表格 | `record_form` | 表格模板和 schema |
| 参考文件 | `reference_file` | 现用或参考程序文件，不一定直接发布 |

## 3. 通用处理链路

```text
原始文件
-> 规范化命名
-> 固定位置归档
-> 文本抽取
-> Markdown 化
-> 拆分内容块
-> 建立块级追溯
-> 人工复核
-> 发布结构化文件
-> 渲染系统包并归档
```

对应表：

| 环节 | 数据表 |
|------|--------|
| 文件归档 | `qms_document_assets` |
| 文件主记录 | `qms_structured_documents` |
| 内容块 | `qms_document_blocks` |
| 块级追溯 | `qms_document_block_links` |
| 修改记录 | `qms_document_change_logs` |

## 4. 外部依据结构化

外部依据的结构化重点是保留来源和形成条款库。

处理步骤：

1. 在 **外部依据** 页面登记或上传文件。
2. 系统记录依据编号、名称、版本、生效日期和附件信息。
3. 维护查新字段：查新日期、结论、证据来源、下次查新日期、查新状态。
4. 点击抽取条款，写入 `qms_clauses` 和 `qms_clause_texts`。
5. 外部依据也会形成 `external_basis` 结构化文件，用于系统包和块级证据。

注意：

- 外部依据原文不直接变成体系要素。
- 条款编号属于 `qms_clauses.clause_number`。
- 查新证据应能回到官网、公告、文件封面或人工复核记录。

## 5. 质量手册结构化

质量手册从现用版本构建框架，后续以结构化内容为维护对象。

推荐流程：

1. 将现用质量手册原文件作为受控文件或结构化资产归档。
2. 生成 `quality_manual` 结构化文件。
3. 按章节拆分 `qms_document_blocks`。
4. 章节编号写入 `qms_manual_sections.section_number`，不是要素编号。
5. 通过 `qms_manual_sections.element_id` 指向对应要素。
6. 手册章节中的控制要求块通过 `qms_document_block_links` 关联条款、要素和程序文件。
7. 修改后发布结构化文件，并通过系统包渲染输出。

质量手册的作用是说明体系总体安排，不应把程序文件的所有操作细节塞回手册。

## 6. 程序文件结构化

程序文件用于把要素和条款要求落到流程、职责、记录和证据。

推荐流程：

1. 以当前质量手册、现用程序文件和参考程序文件为基础。
2. 每个程序文件生成 `procedure` 结构化主记录。
3. 拆分为目的、范围、职责、流程步骤、控制要求、记录要求等块。
4. 对职责块映射岗位。
5. 对记录要求块映射记录表格模板。
6. 对流程或控制要求块映射运行模块。
7. 对参考程序文件建立人工匹配记录，决定是否采纳到正式程序。

常见块类型：

| 块类型 | 用途 |
|--------|------|
| `purpose` | 文件目的 |
| `scope` | 适用范围 |
| `responsibility` | 职责 |
| `process_step` | 流程步骤 |
| `control_requirement` | 控制要求 |
| `record_requirement` | 记录要求 |

## 7. 作业指导书结构化

作业指导书用于承接更细的操作要求。当前模型已预留 `work_instruction` 角色。

建议后续处理方式：

- 先关联到对应程序文件和要素。
- 对关键步骤、设备条件、环境条件、判定规则等建立内容块。
- 与记录表格、设备模块、培训能力模块建立追溯。
- 作业指导书的编号仍属于文件控制，不属于体系要素。

## 8. 记录表格结构化

记录表格结构化有两层：

1. 文件层：表格作为 `record_form` 结构化文件，保留来源和 Markdown。
2. schema 层：字段进入 `record_form_templates.field_schema`，用于运行填写和证据输出。

记录表格的 schema 应尽量从程序文件的记录要求和表格原件共同确认，不能只靠表头猜测。

## 9. 块级追溯

`qms_document_block_links` 是结构化追溯的核心。

一个内容块可以关联多个对象：

- `element_id`：承接哪个体系要素。
- `clause_id`：依据哪个外部条款。
- `manual_section_id`：对应哪个手册章节。
- `procedure_document_id`：关联哪个程序文件。
- `record_form_template_id`：要求或生成哪个记录表格。
- `position_id`：由哪个岗位负责。
- `business_module_id`：由哪个系统模块产生运行证据。

`relation_type` 用于说明关系性质：

| 类型 | 说明 |
|------|------|
| `basis` | 依据来源 |
| `implements` | 实施或承接 |
| `mentions` | 提及 |
| `responsible` | 职责 |
| `requires_record` | 要求形成记录 |
| `renders_to` | 渲染或输出到 |
| `supporting` | 支撑 |

`confidence=review_required` 表示需要人工复核。

## 10. 结构化修改与发布

结构化文件支持局部块编辑。推荐规则：

- 修改前先查看块级追溯，确认影响对象。
- 修改时填写修订说明。
- 修改后写入 `qms_document_change_logs`，记录旧哈希、新哈希、摘要和追溯快照。
- 发布结构化文件后再渲染系统包。
- 系统包归档应保留必要历史，开发运行产物可按运维说明裁剪。

## 11. 系统包输出

系统包用于把当前结构化结果组合为完整体系文件输出。

入口：

```text
/planning/structures/package
```

主要内容包括：

- 外部依据清单和查新状态。
- 质量手册。
- 程序文件。
- 作业指导书。
- 记录表格 schema 文档。
- 块级追溯关系。
- 变更影响摘要。

当前输出路径：

```text
jewelry-qms/runtime/qms_structured/system_package/qms_system_package.md
```

归档路径：

```text
jewelry-qms/runtime/qms_structured/system_package/archive/
```

`runtime/` 属于运行产物，默认不纳入 Git。正式归档策略应由实验室另行确认。

## 12. 复核重点

每次结构化或重渲染后至少复核：

- 文件角色是否正确。
- 文件编号、标题、版本是否与现用文件一致。
- 内容块是否缺少关键章节。
- 条款、要素、记录、岗位、模块映射是否有证据。
- `review_required` 链接是否处理。
- 记录表格 schema 是否覆盖程序文件要求。
- 系统包是否能重新打开且不出现空章节或重复文件。
