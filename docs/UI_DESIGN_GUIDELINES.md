# Jewelry QMS UI 设计规范

## 1. 适用范围

本文档用于规范 `jewelry-qms` 二次开发阶段的界面设计、中文文案和交互行为。规范适用于工作台、体系文件、审批流程、体系策划、记录表格、资源管理、质量改进、系统设置等服务端渲染页面。

当前项目采用 ThinkPHP 8、Bootstrap 5、少量项目样式 `public/static/css/qms.css`。本规范优先兼容现有页面结构，不要求引入新的前端框架。

## 2. 设计目标

界面风格应接近 Apple 式的简洁、克制和清晰，但必须服务于企业质量管理系统的实际工作密度。

核心目标：

- 页面重点清晰：用户一眼知道当前模块、当前状态和下一步操作。
- 中文友好：所有主流程文案使用自然、准确、简短的中文。
- 交互友好：表格、表单、审批、筛选、状态反馈都要可预测、可恢复、少打扰。
- 业务可信：视觉上安静、稳定、精确，避免营销化、炫技化和过度装饰。
- 渐进落地：先统一设计令牌和基础组件，再随功能模块重构逐步应用。

## 3. 总体风格原则

### 3.1 清爽而非空洞

页面应减少装饰和噪音，但不能为了留白牺牲业务信息密度。质量管理系统的核心是文件、状态、责任人、时间、审批和追溯关系，因此列表、表单和详情页必须保持高可读性。

### 3.2 克制而非单调

主界面以白色表面、浅灰背景和蓝色主操作为基础。颜色主要用于状态、风险、当前选择和主要操作，不用于大面积装饰。

### 3.3 重点明确

每个页面只能有一个最主要操作，例如“新建文件”“保存草稿”“提交审核”。次要操作使用描边按钮或文本链接。危险操作必须有明显颜色和二次确认。

### 3.4 中文优先

除 CAPA、CMA/CNAS 等行业通用缩写外，界面应优先使用中文。按钮、状态、提示、空状态、错误信息都应避免技术化表达。

## 4. 设计令牌

### 4.1 色彩

基础色：

- 页面背景：`#F5F5F7`
- 主表面：`#FFFFFF`
- 次级表面：`#FAFAFC`
- 主文字：`#1D1D1F`
- 次级文字：`#6E6E73`
- 弱提示文字：`#86868B`
- 主边框：`#D2D2D7`
- 弱边框：`#E5E5EA`

品牌与交互色：

- 主色：`#0071E3`
- 主色悬停：`#0066CC`
- 主色浅背景：`#EAF4FF`
- 焦点外环：`rgba(0, 113, 227, 0.24)`

语义色：

- 成功：`#34C759`
- 成功浅背景：`#E8F8EE`
- 警告：`#FF9F0A`
- 警告浅背景：`#FFF4DE`
- 危险：`#FF3B30`
- 危险浅背景：`#FFECEB`
- 信息：`#5AC8FA`
- 信息浅背景：`#EAF8FF`
- 中性：`#8E8E93`
- 中性浅背景：`#F2F2F7`

使用要求：

- 主色只用于主按钮、当前导航、链接和关键聚焦状态。
- 危险色只用于删除、作废、驳回、超期等高风险状态。
- 不使用大面积渐变背景。
- 不使用高饱和紫色、厚重深蓝、棕橙等偏营销或偏消费类主题。

### 4.2 字体

字体栈：

```css
-apple-system, BlinkMacSystemFont, "PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif
```

字号建议：

- 页面标题：`22px`，字重 `600`
- 一级区域标题：`18px`，字重 `600`
- 卡片标题：`16px`，字重 `600`
- 正文：`14px`，字重 `400`
- 表格正文：`13px`，字重 `400`
- 表单标签：`13px`，字重 `500`
- 辅助说明：`12px`，字重 `400`

行高建议：

- 标题：`1.3`
- 正文：`1.55`
- 表格：`1.45`
- 按钮和控件：`1.2`

中文排版要求：

- 不使用负字距。
- 不使用过细字重承载关键业务信息。
- 表格和表单字段不得小于 `13px`。
- 错误、审批意见、缺口说明、上传限制等信息必须可读。

### 4.3 间距

基础间距刻度：

- `4px`：图标与文字、紧凑元素内部间距
- `8px`：小控件间距、按钮组间距
- `12px`：表单项内辅助说明、状态标签间距
- `16px`：表单项、卡片内基础间距
- `24px`：页面区域间距
- `32px`：主要区块间距

页面要求：

- 列表页筛选区与表格之间保持 `16px` 间距。
- 卡片之间保持 `16px` 或 `24px` 间距。
- 不使用巨大留白制造空洞感。

### 4.4 圆角和阴影

- 小控件圆角：`6px`
- 卡片圆角：`8px`
- 弹窗圆角：`10px`
- 不使用过度圆润的大圆角。
- 默认卡片使用轻边框，不使用厚重阴影。
- 阴影只用于弹窗、下拉层、悬浮反馈。

## 5. 页面布局

### 5.1 应用外壳

当前主布局位于 `jewelry-qms/app/view/layout/main.html`。后续优化时应保持以下结构：

- 顶部品牌区展示系统名称、当前用户、通知入口和账号操作。
- 主导航按业务域分组：工作台、文件控制、体系策划、审核与评审、质量改进、资源管理、系统设置。
- 导航文案必须使用中文业务词，不使用内部控制器名。
- 当前模块应有清晰选中态。
- 移动端导航必须可折叠，不出现横向溢出。

### 5.2 页面头部

每个业务页面建议包含统一页面头部：

- 页面标题，例如“体系文件列表”。
- 可选说明，例如“管理质量手册、程序文件、作业指导书和记录表格”。
- 右侧主操作，例如“新建文件”。

标题区应替代零散的卡片标题，避免用户进入页面后只看到一个普通卡片。

### 5.3 内容容器

内容容器应遵循：

- 列表、详情、表单使用白色主表面。
- 不在卡片里再嵌套卡片。
- 长表格优先使用全宽容器和横向滚动。
- 工作台可使用统计卡片，但数量和颜色必须克制。

## 6. 组件规范

### 6.1 按钮

按钮层级：

- 主按钮：蓝底白字，用于页面主要动作。
- 次按钮：灰色或蓝色描边，用于返回、重置、下载、查看。
- 危险按钮：红色或红色描边，用于删除、作废、驳回。
- 文本链接：用于低风险跳转。

按钮文案：

- 使用动词开头：新建、保存、提交审核、发起修订、返回列表、下载附件。
- 避免“操作”“处理”“确定”等含糊词。
- 表单保存使用“保存”或“保存草稿”，审批使用“提交审核”或“提交审批意见”。

交互要求：

- 所有按钮必须有 hover、focus、disabled 状态。
- 提交中按钮应禁用，并显示“保存中...”或“提交中...”。
- 按钮组之间保持 `8px` 间距。

### 6.2 表单

表单布局：

- 普通新增和编辑页桌面端可使用两列。
- 复杂记录表格、审批意见、长文本表单优先单列。
- 字段分组应有清晰标题，例如“基本信息”“审批设置”“附件信息”。

字段要求：

- 必填字段使用红色星号。
- 占位符用于示例，不替代标签。
- 帮助文字短句化，例如“支持 PDF、Word、Excel，单个文件不超过 20MB”。
- 错误提示贴近字段，并使用明确中文。

底部操作：

- 右侧固定顺序：取消、保存。
- 审批类表单可使用：驳回、同意。
- 危险动作不得和主保存动作混在一起。

### 6.3 表格

表格是本系统的核心信息载体，默认不改成卡片列表。

表格要求：

- 表头使用浅灰背景。
- 行高保持紧凑但不拥挤。
- 关键字段靠左，例如文件编号、标题、设备编号、人员姓名。
- 数字、日期、状态字段保持对齐。
- 操作列靠右，宽度稳定。
- 空状态使用完整中文说明。

常见操作顺序：

1. 查看
2. 编辑
3. 提交或发起
4. 下载
5. 删除、作废或驳回

### 6.4 筛选区

筛选区应位于列表上方，默认紧凑。

要求：

- 常用筛选项优先展示。
- 搜索框 placeholder 明确写出可搜索字段，例如“搜索文件编号/标题”。
- 按钮顺序为“筛选”“重置”。
- 筛选项超过 5 个时，后续可做“更多筛选”折叠。

### 6.5 状态标签

状态标签的文案应优先来自 `jewelry-qms/config/qms.php` 中的 `statusLabels`。

统一语义：

- 草稿、未开始：灰色
- 审核中、处理中、进行中：蓝色或琥珀色
- 已批准、已发布、已完成、合格：绿色
- 待处理、待整改、缺口、待评价：琥珀色
- 已逾期、已驳回、作废、黑名单：红色或深灰

要求：

- 同一状态在不同模块中颜色含义必须一致。
- 标签文字应短，不使用长句。
- 不用颜色作为唯一信息，必须保留中文文案。

### 6.6 提示和反馈

提示类型：

- 成功：保存成功、提交成功、导入完成。
- 警告：资料缺失、即将到期、存在缺口。
- 错误：保存失败、权限不足、校验不通过。

文案示例：

- “保存失败，请检查必填项。”
- “该文件已提交审核，暂不能编辑。”
- “暂无到期设备。”
- “审批意见已提交。”

要求：

- 成功提示不打断流程。
- 错误提示必须说明用户可以做什么。
- 删除、作废、驳回、提交审核必须二次确认。

### 6.7 空状态

空状态必须写明当前为空的对象和建议动作。

推荐文案：

- “暂无体系文件，可点击右上角新建文件。”
- “暂无审批记录。”
- “暂无到期设备。”
- “暂无已发布体系要素，请先发布现用质量手册附录候选。”

空状态不使用大插画或装饰图，保持业务系统的简洁感。

## 7. 典型页面规范

### 7.1 工作台

工作台用于提醒用户当前最重要的事项，而不是展示全部数据。

应包含：

- 关键待办：待审核文件、超期 CAPA、到期校准、未关闭审核发现。
- 风险优先：超期和缺口信息优先展示。
- 快捷入口：文件控制、设备台账、CAPA、通知中心。

不建议：

- 大面积彩色统计卡片。
- 没有行动价值的指标堆叠。
- 过多图表。

### 7.2 列表页

列表页应保持扫描效率。

结构：

1. 页面头部
2. 筛选区
3. 表格
4. 分页

验收要点：

- 主操作在页面头部右侧。
- 表格首列和标题列清晰。
- 状态颜色统一。
- 操作列不换行、不拥挤。

### 7.3 详情页

详情页用于确认事实和执行下一步动作。

结构：

1. 页面头部：标题、状态、主要操作。
2. 基本信息：使用定义列表或信息表格。
3. 业务过程：审批流程、修订历史、关联记录。
4. 附件和下载。

要求：

- 状态和版本信息必须靠近标题。
- 审批流程要突出当前待处理节点。
- 历史记录使用表格或时间线，不使用零散卡片堆叠。

### 7.4 表单页

表单页用于稳定录入，不追求视觉花哨。

要求：

- 字段分组清晰。
- 必填项明确。
- 长表单保留保存按钮可见性，后续可考虑底部吸附操作栏。
- 保存失败后保留用户已填写内容。

### 7.5 审批流程

审批是高风险交互，应清楚表达：

- 当前节点
- 审批角色
- 审批人
- 审批状态
- 审批意见
- 审批时间

待当前用户处理的节点应有轻微强调。驳回必须要求填写意见。

### 7.6 记录表格

记录表格包含动态字段和可重复表格，是系统中最容易失控的界面。

要求：

- 表格控件保持紧凑。
- “增加一行”“删除”按钮位置固定。
- 删除行要有明确可撤销或确认策略。
- 人员选择器应支持扫描，不应过度卡片化。
- 错误提示必须定位到具体字段或行。

## 8. 中文文案规范

### 8.1 按钮

推荐：

- 新建文件
- 保存
- 保存草稿
- 提交审核
- 发起修订
- 下载附件
- 返回列表
- 筛选
- 重置

避免：

- 操作
- 处理
- 确定
- OK
- Go
- Submit

### 8.2 状态

状态词应短、稳定、可复用。

推荐：

- 草稿
- 审核中
- 已批准
- 已发布
- 已作废
- 待处理
- 整改中
- 已验证
- 已关闭
- 已逾期

### 8.3 提示

提示要说明结果和下一步。

推荐：

- “保存成功。”
- “提交成功，请等待审核。”
- “保存失败，请检查必填项。”
- “当前账号无权执行此操作。”
- “该记录已被引用，不能删除。”

## 9. 响应式要求

桌面端：

- 优先保证表格扫描效率。
- 页面最大宽度可适当放宽，不强制窄容器。
- 导航和操作区不应挤压标题。

平板和小屏：

- 导航折叠。
- 表格允许横向滚动。
- 筛选项纵向排列。
- 操作按钮换行后仍保持顺序。

手机端：

- 首要目标是可读、可点、无溢出。
- 不强制把复杂表格改成卡片，必要时横向滚动。
- 表单单列展示。

## 10. 可访问性和可用性

要求：

- 所有表单控件必须有可见标签。
- 聚焦状态必须可见。
- 颜色不能作为唯一信息。
- 按钮文字必须描述动作。
- 禁用状态要解释原因，至少通过上下文提示表达。
- 关键确认弹窗必须使用明确中文。

## 11. 当前阶段落地顺序

第一阶段：设计令牌与基础组件

- 在 `public/static/css/qms.css` 中整理颜色、字体、间距、按钮、卡片、表格、状态标签。
- 保持 Bootstrap 5 兼容。
- 不引入新前端框架。

第二阶段：应用外壳

- 优化 `app/view/layout/main.html` 的顶部区域、导航层级、通知入口和用户操作。
- 保持现有业务分组。
- 增加当前模块选中态。

第三阶段：三类基础页面

- 列表页
- 详情页
- 表单页

先覆盖体系文件、审批流程、工作台，再扩展到设备、培训、CAPA、体系策划、记录表格。

第四阶段：最终视觉精修

- 等主要功能重构稳定后，再统一检查细节。
- 检查移动端、边界状态、空状态、错误状态和长中文文案。

## 12. 页面验收清单

每个页面通过 UI 规范前，应至少满足：

- 第一眼能看出当前页面是什么。
- 页面只有一个最主要操作。
- 中文文案自然、准确、无中英混乱。
- 表格可扫描，操作列不拥挤。
- 表单字段分组清楚，错误提示贴近字段。
- 状态颜色和文案全系统一致。
- 空状态能告诉用户下一步。
- 危险操作有二次确认。
- 手机宽度下不溢出、不遮挡、不出现按钮挤压。
- 页面未引入不必要装饰、渐变、大圆角或过重阴影。

## 13. 不推荐做法

- 为了“高级感”大量留白，导致业务信息不可见。
- 把数据表格改成卡片瀑布流。
- 每个模块单独定义状态颜色。
- 使用大面积深色导航压迫内容。
- 在业务系统中加入营销式 hero、装饰插画、无意义图表。
- 使用英文按钮或技术错误信息。
- 在功能尚未稳定时做全量换皮。

## 14. 维护规则

- 新增模块必须优先复用本规范中的按钮、表格、表单、状态标签和提示。
- 新增状态文案应先进入配置或统一映射，不在模板中随意硬编码。
- 若业务需要新增组件，应先补充本规范，再落地页面。
- 每次功能重构完成后，应顺手检查对应页面是否符合本规范。
