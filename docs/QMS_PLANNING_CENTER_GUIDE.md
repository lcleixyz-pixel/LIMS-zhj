# 体系策划中心说明

本文档说明 `jewelry-qms` 当前体系策划中心的业务目标、页面入口、核心流程和使用边界。它适用于质量负责人、体系文件管理员、开发维护人员和后续智能体协作。

## 1. 核心目标

体系策划中心用于建立实验室质量管理体系的策划与追溯骨架。当前主链路为：

```text
外部依据 -> 条款库 -> 无编号体系要素 -> 质量手册章节 / 程序文件 / 作业指导书
        -> 记录表格 / 运行模块 / 岗位职责 -> 运行证据 -> 智能体建议
```

这里的关键设计原则是：

- 外部依据的编号只属于条款，例如 `GB/T 27025` 的 `6.2`、`7.8`。
- 体系要素不带业务编号，只保留不可见稳定 `key` 和用户可见中文名称。
- 质量手册章节独立建模，章节编号属于手册章节，不属于体系要素。
- 程序文件、记录表格、运行模块和岗位职责都围绕要素建立追溯关系。
- 智能体只写建议、缺口和证据，不直接修改正式体系数据。

## 2. 页面入口

登录后从顶部导航进入 **体系策划**。

| 页面 | 路径 | 用途 |
|------|------|------|
| 策划中心 | `/planning/index` | 总览覆盖情况、缺口和智能体建议 |
| 要素管理 | `/planning/elements` | 维护无编号体系要素，查看要素详情和覆盖情况 |
| 外部依据 | `/planning/sources` | 登记标准、准则、补充要求并维护查新状态 |
| 条款库 | `/planning/clauses` | 查看和映射正式条款 |
| 文件结构化 | `/planning/structures` | 管理外部依据、手册、程序、记录表格等结构化结果 |
| 追溯矩阵 | `/planning/traceability` | 横向查看要素到文件、记录、岗位、模块的覆盖 |
| 质量方针目标 | `/planning/objectives` | 管理质量方针和质量目标候选内容 |

权限由 `jewelry-qms/config/qms.php` 的 `permissions` 控制。`admin` 拥有全部权限，`quality_manager` 默认拥有策划中心相关权限。

## 3. 推荐初始化顺序

从空库或重建库开始时，推荐按以下顺序执行：

1. 进入 **策划中心**，点击“初始化策划骨架”。
2. 进入 **外部依据**，点击初始化或登记当前有效依据。
3. 对外部依据执行查新，填写查新日期、结论、证据来源和下次查新日期。
4. 从外部依据抽取正式条款，进入 **条款库**。
5. 基于 `GB/T 27025 / ISO 17025` 初始化无编号要素，并建立主条款映射。
6. 将 `CMA`、`CNAS`、`A015` 等补充条款映射到已有要素；确无归属时新增本地补充要素。
7. 进入 **文件结构化**，生成质量手册、程序文件、参考程序文件、记录表格的结构化内容。
8. 人工复核块级追溯、记录表格 schema 和智能体建议。
9. 渲染系统包，形成可归档的完整体系文件输出。

初始化脚本和服务方法按幂等方式设计，重复执行不应产生重复核心数据。

## 4. 外部依据管理

外部依据表示正式的外部规则来源，包括：

- `GB/T 27025 / ISO 17025`
- `CNAS-CL01`
- `CMA` 相关要求
- `CNAS-CL01-A015`
- 其他行业或机构补充文件

业务处理顺序为：

```text
上传或登记依据文件
-> 规范化命名并归档
-> 提取文本和元数据
-> 维护查新状态
-> 抽取条款
-> 映射要素
```

外部依据本身不等于体系要素。依据中的编号和条款内容进入 `qms_sources`、`qms_clauses` 和 `qms_clause_texts`。

## 5. 条款库管理

条款库承载所有外部依据条款。条款编号只存放在 `qms_clauses.clause_number`，例如：

| 来源 | 条款编号 | 说明 |
|------|----------|------|
| GB/T 27025 | `6.2` | 人员要求 |
| GB/T 27025 | `7.8` | 结果报告 |
| CNAS / CMA / A015 | 对应编号 | 补充或特殊要求 |

条款可通过 **条款库** 页面映射到已有要素，也可在确无归属时创建本地补充要素。补充要素仍不显示条款编号，只显示中文名称。

## 6. 无编号要素管理

要素是体系概念节点，例如：

- 人员
- 设备
- 方法确认
- 记录控制
- 内部审核
- 管理评审
- 结果报告

要素用于承接条款、文件、岗位、记录和运行模块的交叉追溯。它不是章节，不是条款，也不是文件编号。

要素列表默认展示：

- 要素名称
- 类型
- 适用性
- 外部条款
- 手册章节
- 程序文件
- 记录表格
- 运行模块
- 岗位职责
- 状态

默认排序优先使用主 `27025` 条款顺序。没有主条款的本地补充要素按 `sort_order` 排在相关要素后。

## 7. 文件结构化

文件结构化将现有文件转换为可追溯、可编辑、可组合的 Markdown 内容块。当前支持的文件角色为：

| 角色 | 说明 |
|------|------|
| `external_basis` | 外部依据 |
| `quality_manual` | 质量手册 |
| `procedure` | 程序文件 |
| `work_instruction` | 作业指导书 |
| `record_form` | 记录表格 |

结构化结果由三层组成：

```text
qms_document_assets          原始文件归档和文本抽取信息
qms_structured_documents     结构化文件主记录
qms_document_blocks          Markdown 内容块
qms_document_block_links     块级追溯关系
```

块级追溯可以指向要素、外部条款、手册章节、程序文件、记录表格、岗位和运行模块。这样后续修改单个块时，可以判断影响哪些条款、文件和证据。

## 8. 记录表格 schema

记录表格模板由 `record_form_templates` 管理，核心字段包括：

- `doc_number`：记录表格编号。
- `name`：记录表格名称。
- `module`：归属业务模块说明。
- `element_id`：主归属体系要素。
- `procedure_doc_id`：来源或要求对应的程序文件。
- `field_schema`：结构化字段 schema。
- `review_status`：schema 复核状态。

记录表格的构建原则是：字段 schema 必须来自程序文件或表格原件的真实要求。智能体可以提示字段缺口，但需要人工复核后才能进入正式模板。

## 9. 追溯矩阵

追溯矩阵按要素横向聚合覆盖情况，关注：

- 外部条款是否已映射。
- 质量手册章节是否已覆盖。
- 程序文件是否已承接。
- 记录表格是否能形成证据。
- 运行模块是否能生成运行数据。
- 岗位职责是否明确。
- 是否存在智能体建议或人工待处理缺口。

常用抽样验证要素包括：

- 人员
- 设备
- 结果报告
- 内部审核
- 管理评审

## 10. 智能体建议边界

智能体建议存放在 `qms_agent_suggestions`，类型包括：

- `gap`：覆盖缺口。
- `mapping`：条款或文件映射建议。
- `document`：文件结构建议。
- `record`：记录表格建议。
- `module`：运行模块建议。
- `responsibility`：岗位职责建议。

建议状态为 `open`、`accepted`、`rejected`。采纳建议不等于自动改正式数据，正式数据仍应通过页面操作、人工复核或受控脚本写入。

## 11. 当前边界

当前系统已经具备策划与追溯骨架，但仍需要业务侧持续补充：

- 外部依据文件的正式查新证据。
- 质量手册现用版本与结构化章节的逐章校对。
- 程序文件与记录要求的逐项确认。
- 记录表格字段 schema 的人工复核。
- 岗位职责与实验室真实组织架构的匹配。
- 运行模块与真实运行证据的持续对账。
